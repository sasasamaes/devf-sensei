# Issues y colaboración

La comunicación es clave en open source.

---

## ¿Qué es un Issue?

Un issue es una tarea, bug o mejora reportada.

```
🐛 Bug → algo no funciona
✨ Enhancement → mejora o feature nueva
📖 Documentation → falta documentación
❓ Question → duda sobre el proyecto
```

Note:
Issues son el centro de comunicación del proyecto.
Todo el mundo puede crear issues.

---

## Labels

Labels organizan los issues:

```
🟢 good first issue → buen punto de partida
🟡 help wanted → necesita ayuda
🔵 enhancement → mejora
🔴 bug → error
⚫ documentation → docs
🟣 question → pregunta
🟠 duplicate → ya existe
```

Note:
Los labels ayudan a filtrar y encontrar issues.
good first issue son para quienes empiezan.

---

## Asignarse un issue

Antes de trabajar en un issue:

```
1. Comenta: "Me gustaría trabajar en esto"
2. Espera confirmación
3. El maintainer te asigna
4. ¡A trabajar!
```

¿Por qué? Para que dos personas no trabajen en lo mismo.

Note:
Mostrar cómo comentar en un issue.
Y cómo se ve la asignación.

---

## Cómo crear un issue

Título claro y descriptivo:

```
❌ "No funciona"
✅ "El botón de guardar no responde en móvil"
```

Descripción (para un bug):

```
## Descripción
[qué pasa]

## Pasos para reproducir
1. Ir a...
2. Hacer clic en...
3. ...

## Comportamiento esperado
[qué debería pasar]
```

Note:
Mostrar la plantilla de issue en vivo.
Un issue bien escrito se resuelve más rápido.

---

## Colaborar con otros

En open source no estás solo.

- Puedes comentar en issues de otros.
- Puedes ayudar a revisar PRs.
- Puedes sugerir mejoras.
- Puedes trabajar en equipo.

La comunidad es lo mejor del open source.

Note:
Animar a los estudiantes a interactuar.
Preguntar dudas en los issues, ayudar a otros.
